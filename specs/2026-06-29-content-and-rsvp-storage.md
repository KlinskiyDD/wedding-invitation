## 7. Бизнес-правила / Алгоритмы (если есть)
- Персональная ссылка:
  - формат рекомендованно: `https://<domain>/?guest=<token>#rsvp`;
  - причина: текущий сайт статический, query token не требует динамического Next route;
  - token должен быть случайным, длинным, непредсказуемым;
  - в базе хранить hash token, а не обязательно сырой token.
- RSVP:
  - гость открывает персональную ссылку;
  - клиент вызывает `GET /api/guest?token=...`;
  - Worker проверяет token, возвращает публично допустимые данные: имя/обращение, количество приглашённых мест, текущий статус ответа;
  - гость отвечает на обязательный вопрос `Придёте ли вы?`: `да` / `нет`;
  - если гость отвечает `да`, форма дополнительно спрашивает:
    - `Поедете ли вы на роспись?`: `поеду на роспись` / `приеду сразу на банкет`;
    - `Какой алкоголь предпочитаете?`: варианты из справочника, обязательно включая `не пью`;
    - `Какие продукты вы не едите?`: свободное поле или список + свободный комментарий;
    - количество гостей в рамках лимита, если приглашение рассчитано на несколько человек;
  - клиент отправляет `POST /api/rsvp`;
  - Worker записывает ответ в D1.
- Countdown:
  - целевая дата: `2026-08-22T15:30:00+03:00`;
  - компонент должен корректно рендериться при static export: начальное значение безопасно считать на клиенте после hydration;
  - если дата уже прошла, показывать нейтральный текст `Наш праздник уже состоялся`.
- Фотоблок:
  - реальные фото не выдумывать и не генерировать вместо пары;
  - если пользователь ещё не передал фото, блок можно оставить подготовленным, но не публиковать с чужими/сгенерированными людьми;
  - допустим временный текстовый placeholder в структуре, но без фейковых портретов.
- Таблица гостей:
  - позже пользователь передаёт таблицу;
  - импорт создаёт записи гостей и генерирует индивидуальные ссылки;
  - ссылки можно вернуть отдельным CSV/XLSX.

## 9. Изменения модели данных / состояния
- Рекомендованное хранилище: Cloudflare D1.
- Почему D1:
  - остаёмся в Cloudflare, где уже деплоится сайт;
  - данные структурированные и хорошо ложатся в SQL;
  - Worker получает доступ к D1 через binding;
  - список гостей не попадает в публичный bundle.
- Минимальная схема:

```sql
CREATE TABLE guests (
  id TEXT PRIMARY KEY,
  token_hash TEXT NOT NULL UNIQUE,
  display_name TEXT NOT NULL,
  party_size INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL
);

CREATE TABLE rsvp_responses (
  id TEXT PRIMARY KEY,
  guest_id TEXT NOT NULL REFERENCES guests(id),
  status TEXT NOT NULL CHECK (status IN ('yes', 'no')),
  ceremony_attendance TEXT CHECK (
    ceremony_attendance IN ('ceremony', 'banquet_only') OR ceremony_attendance IS NULL
  ),
  alcohol_preference TEXT,
  food_restrictions TEXT,
  attendee_count INTEGER NOT NULL DEFAULT 0,
  comment TEXT,
  submitted_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

- Таблицу можно расширить после получения реального файла гостей:
  - `phone`;
  - `side`;
  - `plus_one_name`;
  - `dietary_notes`;
  - `invite_group`.
- Справочник алкоголя на первом этапе держать в коде/content config, не в отдельной таблице, если список небольшой:
  - `вино`;
  - `игристое`;
  - `крепкий алкоголь`;
  - `без предпочтений`;
  - `не пью`.

## 12. Риски и edge cases
- Персональные данные:
  - не хранить guest list в клиентском bundle;
  - token должен быть непредсказуемым;
  - не выводить чужие данные по угадываемому ID.
- Диетические ограничения и алкогольные предпочтения являются чувствительными бытовыми данными:
  - показывать их только владельцу базы/организатору;
  - не логировать значения в публичный frontend;
  - не добавлять их в URL.
- Таймер может давать hydration mismatch, если считать время на сервере.
  - Смягчение: считать countdown в client component после mount.
- Фотоблок без реальных фото может выглядеть фальшиво.
  - Смягчение: подготовить компонент, но финально наполнить после передачи фотографий пары.
- Без таблицы гостей нельзя корректно сгенерировать ссылки.
- Google Forms/Sheets проще, но хуже подходит для персонального UX и контроля данных.
- D1 требует Cloudflare database setup и миграций; это больше работы, но лучше совпадает с текущей инфраструктурой.
- Если RSVP должен быть готов очень быстро, можно временно использовать Google Forms, но это отдельный компромисс.

## 18. Альтернативы и компромиссы
- Вариант: Cloudflare D1 + Worker API. **Рекомендован.**
  - Плюсы: в текущей Cloudflare инфраструктуре, приватный guest list, SQL, персональные ссылки, контроль UX.
  - Минусы: нужно создать D1, миграции и Worker API.
  - Почему выбранное решение лучше: текущий деплой уже на Cloudflare Workers Static Assets; D1 естественно добавляет динамическую часть без OpenNext.
- Вариант: Google Forms + Google Sheets.
  - Плюсы: быстрее всего, удобная таблица ответов.
  - Минусы: слабее персонализация, внешний UX, сложнее приватно проверять tokens, зависимость от Google Forms.
  - Когда выбрать: если нужно запустить RSVP за один вечер без разработки backend.
- Вариант: Airtable.
  - Плюсы: удобная админка и API.
  - Минусы: внешний сервис, secrets/API интеграция, ещё одна платформа.
  - Почему не выбран сейчас: D1 лучше совпадает с текущим Cloudflare stack.
